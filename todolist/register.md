# 登录与注册板块设计规范 (纯静态托管单页应用版)

本规范用于指导开发“我的课程空间”系统中的登录与注册功能。系统不使用重度前端框架，采用原生 JS 页面切换（单页应用）配合后端 Express 与 SQLite 数据库，彻底解决首屏图标文字闪烁（FOUC）的问题。

系统路由路径统一为：`/login`

## 一、 前端界面设计（仿 Google 极简扁平风）

登录与注册界面始终保持在屏幕正中央。全站图标放弃字体库，统一使用代码内嵌的 **SVG 矢量图**，保证加载时绝对不闪烁、不乱码。

### 1. 登录界面 (Login View)

- **顶部内容**：展示一个内嵌的 SVG 图标（替代传统图片/字体），下方写有大字标题 `登录` 。
- **输入框区域**：
    
    - **学号输入框**：左侧是一个标准的文本输入框（用于输入 9 位或 12 位纯数字学号），框内有灰色标签提示“学号”；右侧常驻浅灰色固定后缀文本 `@smail.nju.edu.cn`。当鼠标点击进入输入框时，和之前的设计一样，外边框变为高亮蓝色，提示文字自动缩小挪到输入框左上角。
    - **密码输入框**：标准的密码输入框（隐藏输入内容），用于老用户输入密码登录。
- **底部按钮区**：右下角是一个高亮蓝色的圆形矩形按钮 `[ 下一步 ]`（触发登录）；左下角是一个灰色普通文本按钮 `[ 注册 ]`。点击后，页面不刷新，直接在原地切换为注册界面。

### 2. 注册界面 (Registration View)

- **顶部内容**：展示相同的内嵌 SVG 图标，下方大字标题为 `注册`。
    
- **输入框区域**：
    
    - **学号输入框**：结构与登录界面完全一样（左侧输入学号，右侧固定常驻 `@smail.nju.edu.cn` 文字）。
        
    - **设置密码框**：标准的密码输入框，供新用户设置初始密码。密码规范在后面。
    - 确认密码框：再次输入密码。两次输入的密码需要一样。
    - **验证码输入框**：左侧为 6 位数字验证码输入框；右侧拼接一个防连点按钮 `[ 获取验证码 ]`。
- **人机验证区域**：在 `[ 获取验证码 ]` 点击后，自动弹出 **Cloudflare Turnstile 验证框**。验证成功后可以获取验证码。
- **底部按钮区**：右下角常驻高亮蓝色按钮 `[ 注册 ]`；其左边是按钮 `[ 返回登录 ]`（点击切回登录界面）。
- **隐私声明文本**：在注册界面的左下方，常驻一排灰色小字。不提供勾选框，用户点击“提交注册”即视为默认同意：
    > “继续操作即表示您同意 **《用户协议》** 并确认已阅读 **《隐私政策》** 。”（加粗部分用下划线显示，点击可以打开弹窗，以md形式渲染协议。）

# 二、 后端处理逻辑 (REST API)

### 1. 注册

后端接收前端传来的 `学号` 和 `cf-turnstile-response`（人机验证 Token）：

- **学号格式检查**：后端使用正则表达式对输入的学号进行格式核验。学号必须满足以下三种规则之一，否则直接拒绝并提示：注册失败：
	- 1. 本科生
		- 规则：**9 位纯数字**
		- 正则：`^\d{9}$`
	- 2. 研究生（2022 级及以后，现行最新）
		- 规则：**12 位纯数字**
		- 正则：`^\d{12}$`
	- 3. 研究生（2021 级及以前，老格式）
		- 学硕：`^MG\d{8}$`
		- 专硕：`^MF\d{8}$`
		- 博士：`^BH\d{8}$`
- **查重拦截（保护额度）**：根据输入的学号，拼接为 `学号@smail.nju.edu.cn` 的完整邮箱格式。优先查询用户主表，如果发现该学号已经注册过，直接当场返回 `409` 状态码（提示：该邮箱已注册，请直接登录），不再向下发送邮件，防止浪费邮件额度。
- **密码规范校验**：密码不超过 30 位，且仅允许包含大写字母、小写字母和数字，不符则返回 `400`。
- **人机网关确权**：将拿到的人机验证 Token 和当前请求的 IP 地址，转发给 Cloudflare 官方服务器进行后台核验。如果 Cloudflare 返回核验失败，判定为黑客脚本绕过了前端接口，后端立即拒绝，返回 `403` 状态码。
- **限流与下发**：通过以上核验后，接着检查 IP 风控和 60 秒冷却，全部通过则生成 6 位纯数字验证码写进临时表，并调用邮件服务（Resend）向该学号邮箱发送验证邮件。
- **验证码核验**：去验证码临时表查询该邮箱。比对验证码是否一致、是否过了 5 分钟有效期。
- **写入用户表并销毁凭证**：核验通过后，将密码进行加盐哈希（Bcrypt），在用户表中新建一行记录。系统会自动切分学号生成默认昵称（如：纯数字学号生成 `同学_学号前四位`，字母开头学号生成 `同学_学号开头字母加年份`）。随后，立刻删除临时表里的这条验证码记录，确保验证码只能使用一次。
    
### 2. 登录

后端接收前端传来的 `学号` 和 `密码`，自动将学号拼接为完整的学号邮箱格式，然后去检查账号和密码：

- **学号格式检查**：输入的学号必须符合上述三种学号正则表达式规范之一，不符则返回 `400`。
    
- **密码规范校验**：密码不超过 30 位，且仅允许包含大写字母、小写字母和数字。
    
- **一致性核验**：无论该账号是否注册，只要账号在用户表中不存在，或者输入的密码与数据库中的 Bcrypt 密码哈希值不匹配，后端统一模糊返回：`账号或密码错误`，不向前端暴露账号是否存在，确保安全性。比对成功则发放身份 Token 字符串，前端收到后存入 `localStorage.setItem('auth_token', token)` 并完成登录。
## 三、 安全风控与防刷机制

- **Cloudflare Turnstile 验证锁**：双重卡死防御。前端控制未通过核验时置灰 `[ 获取验证码 ]` 按钮；后端在接收发送验证码请求时，强制先质询 Cloudflare 验证接口。以此彻底拦截黑客用自动化脚本高频调用邮件接口。
    
- **蜜罐隐藏输入框**：在 HTML 代码里故意放一个隐藏的文本框，通过 CSS 让其彻底隐形（用户看不见）。如果是黑客的机器脚本爬虫，会自动往所有输入框里填东西。后端一旦发现这个隐藏框被填了字，直接判定为机器攻击，立刻中断后续处理，并故意返回一个伪造的“成功”响应，迷惑黑客。
    
- **反向代理 IP 获取**：后端代码顶部锁死 `app.set('trust proxy', true);`。确保系统上线通过代理（如 Nginx / CDN）转发后，后端依然能抓到校内学生真正的手机/电脑 IP 地址，防止封禁错 IP。
    

### 接口限流规则表

| **限制维度**    | **处罚阈值**         | **前端交互表现**                               | **后端限流策略**                              |
| ----------- | ---------------- | ---------------------------------------- | --------------------------------------- |
| **单邮箱发送冷却** | 60 秒 / 次，一天至多5次。 | 获取验证码后，按钮变灰并进入 60 秒倒计时，文字变成 `重新发送(60s)`。 | 60 秒内如果对同一个邮箱重复发请求，后端直接拒绝并返回 `429` 状态码。 |
| **验证码暴力盲猜** | 连错 3 次           | 验证码输入框自动清空，提示：“验证码输入错误次数过多，请重新获取”。       | 只要密码/验证码输错 3 次，后端在数据库里**立刻物理删除**该验证码。   |

## 四、 验证码临时表数据库设计 (SQLite)

在 SQLite 中，建立一张独立的临时验证码表 `email_verifications`，与用户主表完全分开，只存临时数据：

SQL

```
CREATE TABLE IF NOT EXISTS email_verifications (
    email TEXT PRIMARY KEY,          -- 自动拼接后的学号邮箱 (如 251234567@smail.nju.edu.cn)
    code TEXT NOT NULL,              -- 6位纯数字验证码
    attempts INTEGER DEFAULT 0,      -- 输错次数计数器 (到 3 次触发销毁)
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
    expires_at DATETIME NOT NULL     -- 过期时间戳 (当前写入时间 + 5分钟)
);
```

- **自动清理过期脏数据**：每次有新同学申请发送验证码时，后端在往这张表写数据的同时，会顺手异步执行一句删除语句：`DELETE FROM email_verifications WHERE expires_at < datetime('now')`。自动清空数据库里所有已经过期的验证码，保持数据库干净，提升查询速度。
