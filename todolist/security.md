# 🔒 风控考虑 — 初步安全分析

> 审查日期：2026-06-02
> 涵盖：全部后端路由、认证体系、文件上传、前端安全
> 状态：初稿，待讨论后实施

---

## 一、认证与鉴权

### 1.1 JWT Secret 硬编码回退值
- **现状**: `routes/middleware/auth.js:4` — `process.env.JWT_SECRET || 'JWT_SECRET_PLACEHOLDER'`
- **风险**: 生产环境若未设置环境变量，所有 token 使用同一密钥。攻击者可直接用该密钥伪造任意用户 token。
- **建议**: 启动时检查 `JWT_SECRET` 是否存在，不存在则拒绝启动并打印错误。密钥至少 32 字节随机字符串。

### 1.2 Token 有效期过长（7天）且无刷新机制
- **现状**: Token 有效期 7 天，无 refresh token，无 token 黑名单。
- **风险**: Token 泄露后攻击窗口长达 7 天。用户修改密码后旧 token 仍有效。无法强制下线。
- **建议**:
  - 缩短 access token 有效期至 2 小时
  - 增加 refresh token（有效期 30 天，存数据库可撤销）
  - 或至少提供"修改密码后使所有 token 失效"机制（在 users 表加 `token_version` 字段，JWT 中包含 version，验证时比对）

### 1.3 登录接口无速率限制
- **现状**: `/api/auth/login` 和 `/api/auth/register` 无任何频率限制。
- **风险**: 暴力破解密码、枚举已注册邮箱、刷验证码短信/邮件。
- **建议**:
  - 登录：同一 IP 每分钟最多 10 次尝试
  - 注册：同一 IP 每小时最多 5 次
  - 验证码发送：同一邮箱每分钟 1 次，每小时 5 次
  - 可用 `express-rate-limit` 或内存计数器实现

### 1.4 验证码强度与防刷
- **现状**: 6 位纯数字验证码，10 分钟有效。`debug_code` 在响应中返回。
- **风险**:
  - 6 位数字只有 100 万种组合，暴力破解成本低
  - `debug_code` 字段在生产环境若未移除，等于明文暴露验证码
- **建议**:
  - 生产环境移除 `debug_code` 响应字段（用环境变量控制）
  - 验证码错误 5 次后锁定该邮箱 15 分钟
  - 考虑增加验证码长度或加入字母

### 1.5 密码策略过于宽松
- **现状**: 仅要求 `password.length >= 6`，无复杂度校验。
- **风险**: 用户可设置 `123456`、`aaaaaa` 等弱密码。
- **建议**: 至少要求包含字母+数字，或引入密码强度检测库（如 `zxcvbn`）。

## 二、输入校验与注入防护

### 2.1 缺少全局请求体大小限制
- **现状**: `app.use(express.json())` 未设置 `limit` 参数（默认 100KB）。
- **风险**: 攻击者发送超大 JSON 体耗尽内存。
- **建议**: `app.use(express.json({ limit: '10kb' }))`

### 2.2 SQL 注入风险点（低概率但需警惕）
- **现状**: 大部分查询使用参数化 `?` 占位符，安全性良好。但有几处字符串拼接：
  - `server.js:109` — `PRAGMA table_info(${table})`（硬编码调用，暂无风险）
  - `routes/search.js` — LIKE 参数使用 `'%${keyword}%'` 通过参数化传入，安全
- **风险**: 目前无实际注入风险，但如果未来有人在 migrateTable 等函数中传入用户输入就会出问题。
- **建议**: 给 migrateTable 加注释"仅限硬编码调用，禁止传入用户输入"。

### 2.3 XSS 风险
- **现状**:
  - 前端大量使用 `escHtml()` 做输出转义，覆盖良好
  - `auth.js:573` — 搜索历史内联 onclick 中的转义不充分
  - `ui.js:176` — `eval(onchangeStr)` 执行 md-select 的 onchange 回调，如果 onchange 内容被注入则可执行任意代码
- **建议**:
  - 搜索历史改用事件委托绑定，不拼接内联 onclick
  - `eval(onchangeStr)` 改为 `new Function` 或用事件系统替代 eval

### 2.4 Markdown 渲染 XSS
- **现状**: `router.js:62` — `markdownit().render(text)` 默认配置可能允许 HTML 标签通过。
- **风险**: 用户在帖子/评论中注入 `<script>` 或 `<img onerror=...>`。
- **建议**: 配置 markdown-it 时启用 `html: false`（禁止原始 HTML）或使用 DOMPurify 清理输出。

## 三、文件上传安全

### 3.1 文件类型仅靠扩展名校验
- **现状**: `routes/materials.js:37-43` — `fileFilter` 只检查 `path.extname` 是否在白名单中。
- **风险**: 攻击者将恶意文件重命名为 `.pdf` 上传。虽然服务端不执行这些文件，但如果前端用 `Content-Type: image/jpeg` 渲染用户上传的文件，可能触发浏览器解析漏洞。
- **建议**:
  - 增加 MIME 类型检查（`file.mimetype`）
  - 上传图片时用 `sharp` 重新编码（去掉可能的恶意载荷）
  - 响应头设置 `Content-Security-Policy` 和 `X-Content-Type-Options: nosniff`

### 3.2 文件名处理
- **现状**: 文件名用 `Date.now() + randomBytes(6) + ext` 重命名，安全性较好。但 `file.originalname` 在 `materials.js:40` 做了 Latin-1 → UTF-8 转码。
- **风险**: 转码后原始文件名可能包含特殊字符（路径穿越字符 `../`）。
- **建议**: 在 `toAttachmentResponse` 中确保 `file_name` 不包含路径分隔符。实际上当前存入数据库的是 `originalname`，下载时用的是服务端生成的安全文件名，风险可控。

### 3.3 上传目录权限
- **现状**: 上传文件存储在 `uploads/materials/` 和 `uploads/post-attachments/`，通过 express.static 提供 `public/` 但不包含 `uploads/`。
- **风险**: 如果 Nginx 等反代配置不当，可能直接暴露上传目录。
- **建议**: 确保生产环境 Nginx 不对 `uploads/` 开放目录列表，且设置 `Content-Disposition: attachment` 强制下载而非渲染。

## 四、API 安全

### 4.1 无 CORS 配置
- **现状**: 无 `cors` 中间件配置。
- **风险**: 如果前后端分离部署（不同域名），跨域请求会被浏览器拦截。但如果是同域部署（当前架构），则无此问题。
- **建议**: 如未来分离部署，需配置 `cors({ origin: 'https://kedazi.app' })`。

### 4.2 缺少 CSRF 防护
- **现状**: API 使用 JWT Bearer Token 认证，无 CSRF token。
- **风险**: JWT 存储在 localStorage 中，不受 CSRF 攻击（Cookie 才有 CSRF 风险）。但如果 token 被 XSS 窃取，攻击者可直接调用 API。
- **评估**: 当前架构下 CSRF 风险低，主要风险来自 XSS（见 2.3/2.4）。

### 4.3 错误信息泄露
- **现状**: 部分接口返回详细错误信息，如 `'该邮箱已注册，请直接登录'`。
- **风险**: 攻击者可通过注册接口枚举哪些邮箱已注册。
- **建议**: 注册时统一返回"验证码已发送至该邮箱"，不区分"已注册"和"新注册"。或在生产环境模糊化错误信息。

### 4.4 接口越权风险审查
- **现状**:
  - `author_id` / `owner_id` 从 JWT 提取，不从请求体获取 ✅
  - 课程成员列表需登录 + 选课 ✅
  - 帖子删除需是作者 ✅
  - 资料删除需是上传者 ✅
  - 邀约编辑/删除需是创建者 ✅
  - 广场帖子删除需是创建者 ✅
- **潜在问题**:
  - `GET /api/courses/:id/posts` 无需登录即可查看所有帖子内容（含附件下载链接）
  - `GET /api/courses/all` 公开全校课程列表（设计如此，但需确认是否有敏感课程信息）
  - `GET /api/user/:id` 和 `/api/user/:id/profile` 虽有隐私过滤，但 nickname 和 avatar_url 始终可见

## 五、业务风控（防刷/防滥用）

### 5.1 关注系统无限制
- **现状**: 无每日关注上限，无关注总数上限。
- **风险**: 批量注册账号互相关注刷粉丝数；恶意用户大量关注他人造成骚扰。
- **建议**: 每日关注上限 50 人，总关注上限 1000 人。

### 5.2 评论/帖子无反垃圾
- **现状**: 帖子和评论无内容审核、无频率限制、无敏感词过滤。
- **风险**: 发布广告、色情、政治敏感内容。
- **建议**:
  - 短期：加入敏感词过滤（本地词库匹配）
  - 中期：接入内容审核 API（如腾讯云/阿里云内容安全）
  - 新用户发帖/评论频率限制

### 5.3 自习邀约可被滥用
- **现状**: 创建邀约无门槛，可发布虚假地点/时间。
- **风险**: 恶意用户大量发布垃圾邀约；发布不当地点信息。
- **建议**: 每日创建邀约上限 5 个；发布后 5 分钟内可编辑。

### 5.4 交友广场可被刷屏
- **现状**: 发帖无频率限制，7 天自动过期。
- **风险**: 批量注册账号刷屏广场。
- **建议**: 每人同时活跃帖子上限 3 个；每日发布上限 5 个。

### 5.5 签到系统可被脚本刷
- **现状**: `/api/auth/checkin` 无需验证码，每日一次。
- **风险**: 脚本自动签到刷连续天数。
- **评估**: 风险较低（刷签到无实际收益），但如后续加入积分/排行榜则需加强。

### 5.6 反馈接口可被刷
- **现状**: `/api/user/feedback` 无频率限制。
- **风险**: 恶意用户大量提交垃圾反馈。
- **建议**: 每日上限 5 条。

## 六、数据安全与隐私

### 6.1 数据库文件明文存储
- **现状**: `db/eduspace.db` 是 sql.js 导出的 SQLite 文件，明文存储在磁盘。
- **风险**: 服务器被入侵后，攻击者可直接读取全部用户数据（含密码哈希、邮箱、QQ 等）。
- **建议**:
  - 生产环境考虑使用加密 SQLite（如 SQLCipher）
  - 或将数据库迁移到 PostgreSQL/MySQL 并启用磁盘加密
  - 确保 `db/` 目录不在 git 中（当前 `.gitignore` 需检查）

### 6.2 密码哈希安全性
- **现状**: bcryptjs, cost=10。✅ 安全。
- **评估**: cost=10 在当前硬件下约 100ms，足够安全。如需更高安全性可提升至 12。

### 6.3 用户隐私数据暴露面
- **现状**:
  - QQ 号在课程成员列表、广场帖子详情中对已登录用户可见
  - 微信号、抖音号在公开主页中对已登录用户可见
  - 通过 `GET /api/user/:id/profile` 可获取任意用户的公开资料
- **风险**: 爬虫可批量收集用户联系方式用于营销/骚扰。
- **建议**:
  - QQ/微信/抖音等联系方式仅对互相关注的用户可见
  - 或需对方确认后才可见（类似"查看联系方式"按钮）
  - 添加接口频率限制防止批量爬取

### 6.4 通知内容泄露
- **现状**: 通知消息中包含帖子标题、评论内容摘要等。
- **风险**: 如果通知被未授权用户访问（如 token 泄露），可看到敏感内容。
- **评估**: 当前通知列表有 authMiddleware 保护，风险可控。

## 七、前端安全

### 7.1 Token 存储位置
- **现状**: JWT 存储在 `localStorage`。
- **风险**: XSS 攻击可通过 `localStorage.getItem('kedazi_token')` 窃取 token。
- **评估**: 这是 SPA 的常见做法。替代方案是 HttpOnly Cookie（防 XSS 读取），但需 CSRF 防护。当前架构下 XSS 防护是关键。

### 7.2 前端代码无完整性校验
- **现状**: JS 文件通过 `<script type="module">` 加载，无 SRI（Subresource Integrity）。
- **风险**: CDN 或中间人攻击可篡改 JS 文件。
- **建议**: 生产环境构建时为 JS 文件添加 SRI hash。

### 7.3 `eval()` 使用
- **现状**: `ui.js:176` — `eval(onchangeStr)` 执行 md-select 的 onchange 回调。
- **风险**: 如果 `onchangeStr` 内容被注入（如通过 URL 参数构造恶意 select 选项），可执行任意 JS。
- **评估**: 当前 onchange 值都是硬编码的函数名（如 `refreshInvites()`），暂无实际风险。但应改为事件系统。
- **建议**: 用 `container._onchangeCallback = onchangeStr` 配合 `window[onchangeStr]` 调用（限白名单函数），避免 eval。

## 八、运维安全

### 8.1 无 HTTPS 强制
- **现状**: 无 HTTP → HTTPS 重定向，无 HSTS 头。
- **风险**: 中间人攻击窃取 token 和用户数据。
- **建议**: 生产环境 Nginx 配置 HTTPS + HSTS。

### 8.2 无安全响应头
- **现状**: 未设置以下安全头：
  - `X-Content-Type-Options: nosniff`
  - `X-Frame-Options: DENY`
  - `X-XSS-Protection: 0`（现代浏览器推荐关闭，依赖 CSP）
  - `Content-Security-Policy`
  - `Strict-Transport-Security`
- **建议**: 使用 `helmet` 中间件一键配置。

### 8.3 无日志与监控
- **现状**: 仅 `console.log` 输出，无结构化日志，无异常监控。
- **风险**: 被攻击时无法追溯；线上错误无法及时发现。
- **建议**:
  - 接入错误监控（如 Sentry）
  - 关键操作（登录、注册、删除）记录审计日志
  - 异常请求频率告警

### 8.4 `db.save()` 频繁写盘
- **现状**: 几乎每个写操作后都调用 `db.save()`（将整个数据库导出到文件）。
- **风险**:
  - 高并发下可能数据丢失（两次 save 之间的操作未持久化）
  - 频繁全量写盘影响性能
- **建议**: sql.js 的 `save()` 是全量导出，建议改为定时保存（如每 30 秒）或在关键操作后保存。

## 九、紧急预案

### 9.1 账号被盗应急
- 提供"紧急冻结账号"管理员接口
- 用户可自行修改密码使所有 token 失效（需先实现 token version 机制）

### 9.2 数据泄露应急
- 准备用户通知模板
- 准备强制所有用户重新登录的脚本（清除所有 token）

### 9.3 恶意内容应急
- 管理员可快速删除帖子/评论/资料
- 准备封禁用户接口（暂未实现用户状态字段）
